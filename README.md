# Safeguard Mechanism 2024-25 Dashboard — Methodology Notes

**Companion document to** `safeguard_dashboard_v2.html` · **Built** 17 April 2026 · **Data vintage** CER publication 15 April 2026

---

## 1. Why this document exists

The earlier `safeguard_full_analysis.html` contained multiple fabricated and inflated facility-level numbers (Boyne Island Smelter overstated 2.2×, phantom facilities such as "Hay Point Coal Terminal" and "Mount Arthur Coal Mine" that do not appear in the CER table, incorrect headline deltas). This methodology note exists so every figure in the new dashboard is traceable back to a specific cell in a published CER file.

## 3. Column mapping

The CER tables use the following canonical columns (2024-25 CSV header). The dashboard reads each as follows:

| CER column | Dashboard field | Notes |
|---|---|---|
| Facility name | `name` | Normalised (collapse whitespace, en/em-dash → ASCII hyphen) |
| Responsible emitter | `operator` | |
| State/Territory of operation | `state` | |
| ANZSIC | `anzsic` | Used to derive `sector` via grouping rules below |
| ERC | `erc` | Emissions reduction contribution rate — typically 0.902 (standard), 0.941 or 0.98 (TEBA-adjusted) |
| Baseline emissions number | `baseline` | Site-specific baseline in t CO₂-e |
| Covered emissions | `covered` | Scope-1 emissions covered by the Safeguard |
| Borrowing adjustment amount | `borrowing` | Adjustment when a facility elects to borrow from a future year |
| ACCUs issued | `accus_issued` | Safeguard Mechanism Credits (SMCs) are separate |
| ACCUs deemed surrendered | `accus_deemed_surr` | ACCUs auto-surrendered under legacy arrangements |
| ACCUs surrendered | `accus_surr` | Voluntary / compliance-driven surrender |
| SMCs surrendered | `smcs_surr` | Safeguard Mechanism Credits surrendered |
| Net emissions number | `net_emissions` | Covered − (ACCUs + SMCs surrendered + deemed) |
| Net position number | `net_position` | Net emissions − baseline (negative = beat baseline) |
| SMCs issued | `smcs_issued` | SMCs generated by the facility (beat baseline) |
| Cumulative MYMP net emissions / net position | `cum_mymp_*` | Multi-year monitoring period running totals |
| GHG Carbon dioxide / Methane / Nitrous oxide / Other | `co2`, `ch4`, `n2o`, `other_ghg` | Gas-level breakdown |
| Notes | `notes` | Free-text (e.g. "The facility is covered by a landfill baseline") |

Numeric cells in the CER CSV use Australian-locale formatting (`"1,234,567"`) and the literal string `" - "` for null/zero. The parser `clean_num` strips commas/spaces and returns `None` for `-`, `—`, empty strings, or unparseable values. All aggregates treat `None` as zero.

## 4. Threshold & scope classifications

The Safeguard Mechanism applies to facilities with ≥100,000 t CO₂-e Scope-1 covered emissions in a year. The CER published table includes both above- and below-threshold facilities (the latter typically because they have an active baseline but fell below the trigger in that year).

| Category | 2024-25 | 2023-24 |
|---|---|---|
| Rows in CER table | 228 | 233 |
| Above 100,000 t covered (designated large facilities) | **208** | 217 |
| Below 100,000 t covered | 20 | 16 |

**The "above threshold" explorer tab** shows the 208 facilities. **The "secondary" tab** shows:
1. **Fell out** (17): in the 2023-24 table but not in the 2024-25 table at all. 8 of these were above 100k in 2023-24 (red badge); 9 were already below threshold.
2. **Dropped below threshold** (14): in both tables, but covered fell below 100k in 2024-25 (Grosvenor Mine, Nickel West cluster, Yarwun Nitrates, etc.).
3. **All 20 below-threshold rows in 2024-25**: complete list for transparency.

## 6. Sector grouping

ANZSIC classes are grouped into 13 pragmatic sectors for the charts. The mapping is heuristic (string-matching on the ANZSIC class text) and is defined in `dataprep/build_dashboard_data.py::sector_group`:

- Coal mining · Oil & gas · Metal ore mining · Non-ferrous metal mfg · Iron & steel · Cement & lime · Chemicals & fertiliser · Electricity · Waste · Transport & pipelines · Food & ag · Glass · Other

Where a facility's ANZSIC text does not match any specific keyword it falls into `Other`. This is intended for dashboard presentation, not for formal reporting.

## 7. Computed / aggregated numbers

All rollups are simple sums over facility rows with `None` treated as 0:

- `totals_25` / `totals_24`: sum across all rows in each year's CER table.
- `sectors_25` / `sectors_24`: sum within each sector group.
- `top_emitters_25`: sort by `covered` descending, top 15.
- `top_accu_users_25`: sort by `accus_surr + accus_deemed_surr` desc.
- `top_smc_users_25`: sort by `smcs_surr` desc.
- Combined top credit users: sort by `accus_surr + accus_deemed_surr + smcs_surr` desc.
- `top_excess_25`: rows where `covered > baseline`, sorted by excess desc, top 20.
- `top_beaters_25`: sort by `net_position` ascending (most negative = biggest beat), top 20.
- `biggest_risers` / `biggest_fallers`: for facilities with a 2023-24 match, sort by `covered_25 − covered_24`.

## 9. Pipeline (reproducibility)

The dashboard was built by three Python scripts in `/sessions/happy-tender-mendel/dataprep/`:

1. **`load_data.py`** — reads the raw CSV and XLSX, normalises facility names, parses numeric cells, writes `raw_data.json`.
2. **`build_dashboard_data.py`** — applies the rename map, classifies threshold groups, computes aggregates, top-N lists, and YoY matches. Writes `dashboard_data.json` (~362 KB).
3. **`generate_html.py`** — embeds the JSON into the HTML template and writes `safeguard_dashboard_v2.html`.

To regenerate: `python3 load_data.py && python3 build_dashboard_data.py && python3 generate_html.py`.

---

## 10. Verification

After generation, every number in the dashboard's headline KPI cards, narrative cards, and top-N tables was re-checked against the raw CER CSV by extracting the rendered HTML values and diffing against fresh aggregates computed directly from the source file (see the auto-diff log in the verification step of the build).

If any row in the dashboard appears inconsistent with the current CER table, trust the CER table and re-run the pipeline. The rename map is the only hand-curated input and may be amended as additional identity changes are identified.
